Getting Started & Installation

Installation

pip install vipentium

Once installed, you are ready to write tests using the framework's built-in utilities and decorators.

Usage Instructions

Vipentium provides two ways to run your tests. Replace <test_path> with the location (directory, file, or module) containing your tests.

Type Command Example
Recommended vipentium-runner <test_path> [options] vipentium-runner test_example.py --parallel
Alternative python -m vipentium.vipentium_runner <test_path> [options] python -m vipentium.vipentium_runner test_example.py --parallel

For example:
Recommended: vipentium-runner test_example.py --parallel
Alternative: python -m vipentium.vipentium_runner test_example.py --parallel

Detailed Command-Line Options for Vipentium

Vipentium supports a number of useful command‚Äëline options so you can customize how tests are discovered, executed, and reported. Below is an in-depth explanation of each option, including when to use it, what it does, and sample commands.

<test_path> indicates the relative or absolute path to the directory, file, or module containing your tests. Vipentium automatically looks for test files that start with test_. Using the proper path ensures that all tests will be discovered. 

# Running tests in a directory:
vipentium-runner tests/

# Running tests in a specific file:
vipentium-runner tests/test_example.py

# Running tests in a module:
vipentium-runner tests.my_module

--parallel

The --parallel option enables parallel execution of tests using threads. This can significantly reduce the test runtime, especially for test suites with many independent tests. Use it when your tests can be safely executed concurrently. 

# Run tests in parallel to speed up execution:
vipentium-runner tests/ --parallel

Note: Parallel execution is best for tests that do not modify shared state.

--process

The --process option instructs Vipentium to run tests in separate processes rather than threads. This type of isolation can be crucial when tests might interfere with each other or when you need to avoid issues with Python‚Äôs Global Interpreter Lock (GIL). 

# Run tests in separate processes for increased isolation:
vipentium-runner tests/ --parallel --process

Tip: Use --process in combination with --parallel for maximum benefit if you require strict isolation.

--verbose

The --verbose option increases the verbosity of the output. It provides additional details on each test's execution process, making it easier to debug and analyze test behavior. 

# Run tests with detailed output:
vipentium-runner tests/ --verbose

--report-json <filename>

With the --report-json <filename> option, Vipentium generates a JSON report summarizing the test run. This report will include data such as the total number of tests, passed and failed counts, duration, and detailed per-test results. 

# Generate a JSON report and save it as 'report.json':
vipentium-runner tests/ --report-json report.json

This file can then be used for further automated analysis, or integrated into CI/CD pipelines.

--report-html <filename>

The --report-html <filename> option instructs Vipentium to output an HTML report. Unlike the JSON report, the HTML version is human-readable and is presented as a formatted web page that summarizes your test run. 

# Generate an HTML report and save it as 'report.html':
vipentium-runner tests/ --report-html report.html

This is particularly useful when you want to review the test results in a web browser or share the report with team members.

--mark <marker>

The --mark <marker> option filters tests based on a custom marker tag that you have assigned using the @mark decorator. This allows you to run only a subset of tests, such as all integration tests or all tests tagged as "critical." 

# Run only tests marked with "integration":
vipentium-runner tests/ --mark integration

# Run tests tagged with multiple markers:
vipentium-runner tests/ --mark database --mark critical

This selective execution can save time and make debugging faster by targeting problematic or high-priority tests.

Writing Test Cases

Vipentium makes it simple to write tests by following some basic conventions:

  • File Naming: Test files should start with test_ (e.g., test_utils.py).
  • Class Definition: Each test class must inherit from the TestCase class.
  • Method Naming: Test methods should start with test_ so that they are automatically discovered.
  • Assertions: Utilize the built-in assert_equal(a, b) method (or Python‚Äôs assert) to verify expected results.
  • Decorators: Enhance tests using decorators like @parameters, @timeout, @retry, @mark, and @fixture.

Test File Naming

# Correct naming:
test_module.py
test_calculator.py

# Incorrect naming:
module_test.py
Calculator.py

Test Class Definition

from vipentium import TestCase

class TestCalculator(TestCase):
    def test_addition(self):
        self.assert_equal(2 + 2, 4)

    def test_multiplication(self):
        self.assert_equal(3 * 3, 9)

Assertions and Decorators

Use assertions to validate your results. Decorators extend the test's behavior.

# Using the @parameters decorator:
from vipentium import TestCase, parameters

class TestSummation(TestCase):
    @parameters(
      (1, 2, 3),
      (5, 7, 12),
      (10, 15, 25)
    )
    def test_sum(self, a, b, expected):
        self.assert_equal(a + b, expected)
        
# Standard assertions inside tests:
from vipentium import TestCase

class TestConversion(TestCase):
    def test_uppercase(self):
        result = "hello".upper()
        self.assert_equal(result, "HELLO")
        # Native assert can also be used:
        assert result == "HELLO"
        
# Using multiple decorators:
from vipentium import TestCase, timeout, retry

class TestComplexBehavior(TestCase):
    @timeout(3)
    @retry(2)
    def test_with_delay(self):
        import time
        time.sleep(1)
        self.assert_equal(2 * 2, 4)

Fixtures

Fixtures automate setup of test data. They can be function-scoped or session-scoped.

from vipentium import fixture

@fixture(scope="function")
def sample_list():
    return [1, 2, 3]

@fixture(scope="session")
def config_data():
    return {"env": "testing", "version": "1.0"}

Usage Guidelines

Follow these guidelines to make your test suite organized and maintainable:

File Naming Conventions

# Recommended file names:
test_authentication.py
test_api_integration.py
test_database.py

Organizing Test Classes

from vipentium import TestCase

class TestUserManagement(TestCase):
    def test_create_user(self):
        # Code to test user creation
        self.assert_equal(create_user("bob"), True)

    def test_delete_user(self):
        # Code to test user deletion
        self.assert_equal(delete_user("bob"), True)

Consistent Method Naming

class TestMathOperations(TestCase):
    def test_addition(self):
        self.assert_equal(1 + 1, 2)
        
    def test_subtraction(self):
        self.assert_equal(5 - 3, 2)

Example Usage

The following is a comprehensive example that demonstrates the full range of Vipentium features. It includes fixtures, parameterized tests, decorators, and multiple test cases. 

###############################################################################
# Comprehensive Test Suite Example Using Vipentium
###############################################################################

from vipentium import TestCase, parameters, timeout, retry, mark, fixture

# --------------------------
# Fixture Definitions:
# --------------------------
@fixture(scope="function")
def simple_list():
    """A fixture that returns a fresh list for every test."""
    return [1, 2, 3]

@fixture(scope="session")
def shared_resource():
    """A fixture that returns a resource shared across the test session."""
    print("Setting up shared resource...")
    return {"message": "Hello from shared resource"}

# --------------------------
# Basic Operations Test Suite:
# --------------------------
@mark("basic")
class TestBasicOperations(TestCase):
    def test_addition(self):
        self.assert_equal(2 + 2, 4)
    
    def test_string_concat(self):
        self.assert_equal("hello" + "world", "helloworld")
    
    def test_list_length(self, simple_list):
        self.assert_equal(len(simple_list), 3)
    
    def test_shared_message(self, shared_resource):
        self.assert_equal(shared_resource["message"], "Hello from shared resource")

# --------------------------
# Math Functions and Parameterized Tests:
# --------------------------
@mark("math")
class TestMathFunctions(TestCase):
    @parameters(
        (5, 2, 7),
        (10, -3, 7),
        (0, 0, 0)
    )
    def test_add_parameterized(self, a, b, expected):
        self.assert_equal(a + b, expected)
    
    def test_division(self):
        self.assert_equal(10 / 2, 5)
    
    def test_float_equality(self):
        self.assert_equal(3.14, 3.14)
    
    @timeout(3)
    @retry(2)
    @parameters(
        (7, 11, 18),
        (20, 30, 50)
    )
    def test_complex_addition(self, a, b, expected):
        import time
        time.sleep(1)
        self.assert_equal(a + b, expected)

# --------------------------
# List Manipulation Test Suite:
# --------------------------
@mark("list_operations")
class TestListManipulation(TestCase):
    def test_append(self, simple_list):
        simple_list.append(4)
        self.assert_equal(simple_list, [1, 2, 3, 4])
    
    def test_pop(self, simple_list):
        popped = simple_list.pop()
        self.assert_equal(popped, 3)
        self.assert_equal(simple_list, [1, 2])
    
    def test_contains(self, simple_list):
        self.assert_equal(2 in simple_list, True)
        self.assert_equal(5 in simple_list, False)

###############################################################################
# End of Comprehensive Test Suite Example
###############################################################################

This example combines various Vipentium features into one suite, demonstrating how to mix and match decorators, fixtures, and multiple tests.

Extending with Plugins

Vipentium allows advanced customization through plugins. By implementing the Plugin class, you can intercept hooks during test execution to add custom logging, notifications, or other behaviors.

Creating a Custom Plugin

To create a plugin:

  1. Create a class that inherits from Plugin.
  2. Override any hook methods you require, such as before_test, after_test, on_start_suite, and on_end_suite.
  3. Register your plugin using the register_plugin() function.
###############################################################################
# Custom Report Plugin Example
###############################################################################
from vipentium import Plugin, register_plugin

class CustomReportPlugin(Plugin):
    def before_test(self, test_name, test_class, method_name, parameters):
        print(f"[PLUGIN] Starting test: {test_name} ({test_class}.{method_name})")

    def after_test(self, test_name, test_class, method_name, parameters, success, message, duration):
        if success:
            print(f"[PLUGIN] Test '{test_name}' PASSED in {duration:.2f}s")
        else:
            print(f"[PLUGIN] Test '{test_name}' FAILED: {message}")

    def on_start_suite(self):
        print("[PLUGIN] Test suite started.")

    def on_end_suite(self):
        print("[PLUGIN] Test suite ended.")

def load_my_plugins():
    # Register the custom plugin
    register_plugin(CustomReportPlugin())

In your test runner script, load the plugins before executing tests:

# test_runner.py
from vipentium import vipentium_runner
from my_plugin_module import load_my_plugins

# Register plugins
load_my_plugins()

# Run the tests
if __name__ == "__main__":
    vipentium_runner("path/to/your/tests", options=["--parallel"])

Custom plugins give you the flexibility to integrate with external systems, send notifications, or log additional data during testing.
