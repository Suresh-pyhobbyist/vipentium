Installation

Install Vipentium using pip:

Copied!

⚙️ Command-Line Options

Option Description
<test_path> The path to the directory, file, or module where vipentium should discover and run tests.
--parallel Enable parallel test execution using threads for potentially faster test runs.
--process Use separate processes for parallel test execution (requires --parallel). Provides better isolation.
--verbose Enable verbose output, showing more details about the test execution process.
--report-json <filename> Generate a test report in JSON format and save it to the specified filename.
--report-html <filename> Generate a test report in HTML format and save it to the specified filename.
--mark <marker> Only run tests that are decorated with the specified marker. This option can be used multiple times.

✍️ Writing Effective Tests

Test Case Structure

Organize your tests within classes that inherit from vipentium.TestCase. This provides a structured way to group related tests.

  • Test File Naming: Files should start with test_ (e.g., test_utils.py).
  • Test Class Definition: Create classes inheriting from TestCase.
  • Test Method Definition: Methods within test classes must start with test_ (e.g., test_calculate_sum).

Assertions

Verify expected outcomes using assertions. The TestCase class offers assert_equal(a, b) for direct comparison. Standard Python's assert can also be used.

from vipentium import TestCase, parameters, mark, fixture
    
    @mark("basic")
    class TestBasicOperations(TestCase):
        def test_addition(self):
            # Asserts that 2 + 2 equals the expected value of 4
            self.assert_equal(2 + 2, 4)
    
        def test_string_concat(self):
            # Asserts that the concatenation of "hello" and "world" is "helloworld"
            self.assert_equal("hello" + "world", "helloworld")
    
        def test_list_length(self, simple_list):
            # Asserts that the length of the 'simple_list' fixture is 3
            self.assert_equal(len(simple_list), 3)
    
        @mark("math")
        class TestMathFunctions(TestCase):
            @parameters((5, 2, 7), (10, -3, 7), (0, 0, 0), name="addition_examples")
            def test_add_parameterized(self, a, b, expected):
                # Runs the test multiple times with different inputs (a, b) and asserts the 'expected' result
                self.assert_equal(a + b, expected)

Decorators for Enhanced Testing

Decorators provide powerful ways to modify test behavior:

  • @parameters(*args, **kwargs): Executes a test method with multiple input sets. The name argument can provide descriptive names for each run.
  • @timeout(seconds): Sets a maximum execution time for a test, preventing indefinite hangs.
  • @retry(times): Automatically retries a failing test a specified number of times.
  • @mark(*tags): Adds metadata (tags) to tests, allowing for selective execution.
  • @fixture(scope="function"|"session"): Defines reusable setup and teardown logic. "function" scope creates a new fixture instance for each test, while "session" creates it once per test session.
from vipentium import TestCase, parameters, timeout, retry, mark, fixture
    
    @fixture(scope="function")
    def simple_list():
        """Provides a simple list [1, 2, 3] for test methods."""
        return [1, 2, 3]
    
    @fixture(scope="session")
    def shared_resource():
        """Sets up a shared resource once per session and cleans it up."""
        print("\\nSetting up shared resource...")
        data = {"message": "Hello from shared resource"}
        yield data  # Provides the fixture value
        print("\\nTearing down shared resource...")
    
    @mark("slow")
    class TestSlowOperation(TestCase):
        @timeout(1)
        def test_sleep(self):
            # This test will fail if it takes longer than 1 second
            import time
            time.sleep(0.5)
            self.assertTrue(True)
    
    @mark("unstable")
    class TestWithRetry(TestCase):
        counter = 0
        @retry(3)
        def test_flaky_operation(self):
            # This test will be retried up to 3 times if it fails
            self.counter += 1
            if self.counter < 3:
                self.assert_equal(1, 2) # Will fail on the first two attempts
            else:
                self.assert_equal(1, 1) # Will pass on the third attempt

🔌 Extending with Plugins

Vipentium's plugin system allows you to hook into various stages of the test execution process, enabling customization and extension of its behavior.

Creating a Plugin

To create a plugin, define a class that inherits from vipentium.Plugin and override specific hook methods.

  • Inherit from Plugin: Your plugin class must inherit from vipentium.Plugin.
  • Override Hook Methods: Implement methods like before_test (executed before each test), after_test (executed after each test), on_start_suite (executed at the beginning of the test suite), and on_end_suite (executed at the end of the test suite) to perform custom actions.
  • Register Your Plugin: Use the register_plugin() function to make your plugin active.
from vipentium import Plugin, register_plugin
    
    class CustomReportPlugin(Plugin):
        def before_test(self, test_name, test_class, method_name, parameters):
            print(f"[PLUGIN] Starting test: {test_name}")
    
        def after_test(self, test_name, test_class, method_name, parameters, success, message, duration):
            if success:
                print(f"[PLUGIN] Test '{test_name}' passed in {duration:.2f}s")
            else:
                print(f"[PLUGIN] Test '{test_name}' failed: {message}")
    
        def on_start_suite(self):
            print("[PLUGIN] Test suite started.")
    
        def on_end_suite(self):
            print("[PLUGIN] Test suite finished.")
    
    def load_my_plugins():
        register_plugin(CustomReportPlugin())
    
    # Call this function before running your tests to activate the plugin.
    # For example, you might call it in a conftest.py file or directly in your test runner script.
    # load_my_plugins()

📊 Detailed Reporting

Vipentium offers built-in options for generating detailed reports of your test execution, providing insights into test results and performance.

Report Formats

  • JSON Report: A structured .json file containing a comprehensive summary of the test run, including details of each test case, its status (passed, failed), duration, and any error messages. This format is ideal for programmatic analysis.
  • HTML Report: A user-friendly, self-contained .html file that presents the test results in an easily readable format within a web browser. It typically includes summaries, individual test details, and any failure information.

Generating Reports via Command-Line Options

Use the following command-line options with vipentium-runner to generate reports:

  • --report-json <filename>: Specifies the filename for the generated JSON report (e.g., --report-json report.json).
  • --report-html <filename>: Specifies the filename for the generated HTML report (e.g., --report-html report.html).
vipentium-runner test_suite.py --report-json test_results.json
Copied! 
vipentium-runner tests/ --report-html test_report.html
Copied!
